`/cip` 에 **배포 단계**를 더한 스킬이다. `$ARGUMENTS` 는 `/cip` 과 동일하게 해석한다(일반 모드=커밋 메시지 힌트, worktree 모드=대상 브랜치 override).

## 1. 커밋·push (= /cip)

먼저 `cip` 스킬의 전체 흐름을 그대로 수행한다 — 모드 판별, 리모트 도달성 사전 체크, 문서 갱신 판단, 명시적 스테이징, 커밋, push, worktree 모드의 rebase·ff-merge·PR body 갱신·worktree 정리까지 전부 동일하다. 그 규칙을 여기 중복해 적지 않으니 `cip` 스킬(`gemini/skills/cip/SKILL.md`)의 내용을 따른다.

push(또는 worktree 모드의 대상 브랜치 push)가 **실패하거나 건너뛰어졌으면 배포하지 않는다.** 리모트 도달 불가로 push 를 건너뛴 경우, 배포도 하지 않고 그 사실을 보고한다. 판정 기준은 `cip-push.sh` 종료코드 — **0 일 때만** 배포한다.

## 2. 배포

push 가 성공한 뒤에만 배포를 진행한다. **배포 방법은 이 스킬에 미리 적지 않는다 — 실행 시점에 직접 파악하라.**

- 이 repo 의 `AGENTS.md`/`CLAUDE.md`/`README.md`, 그리고 `deploy`·`release`·`Makefile`·CI 워크플로(`.github/`)·배포 스크립트(`script/`, `deploy/` 등)를 읽어 **이 프로젝트의 실제 배포 절차**를 확인한다.
- 명시된 절차가 있으면 그대로 따른다. 배포 전용 스킬(예: 이 repo 에 맞는 deploy-* 스킬)이 있으면 그것을 우선 사용한다.
- 배포 방법을 확정할 수 없으면 **추측해서 실행하지 말고**, 무엇을 찾았고 무엇이 불확실한지 사용자에게 보고한 뒤 어떻게 배포할지 묻는다.
- 배포가 비가역적·외부 영향이 큰 동작(프로덕션 반영, 외부 서비스 발행 등)이면 실행 전에 사용자에게 확인한다. 단, 해당 repo 규칙이 "확인 없이 배포"를 명시하면 그 규칙을 따른다.

배포 명령의 출력을 끝까지 확인해 성공/실패를 판정하고, 결과(배포 대상·버전·URL 등)를 사용자에게 보고한다. 실패하면 자동으로 되돌리거나 재시도하지 말고 즉시 보고·중단한다.
