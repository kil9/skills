추가할 마일스톤: `$ARGUMENTS`

**모드 판별.** repo 루트에 `backlog/` 디렉터리(`backlog/config.yml`)가 있으면 **backlog 모드**,
없으면 **레거시(PLAN.md) 모드**. 어느 모드든 구현·커밋은 하지 않는다 — 마일스톤과 태스크만 만든다.
플랜 파일도 `backlog/` 도 없는 새 저장소면 `/init-backlog` 으로 먼저 초기화하도록 안내한다.

## 규모 판별 (모드 공통 선행)

항목을 서술할 만큼만 가볍게 조사한 뒤, 작업이 **태스크 하나로 끝날 것 같으면** 재량으로 축소하지
말고 `ask_question` 으로 묻는다:

- **단일 태스크로** — `/add-task` 로 위임하고 이 스킬은 종료한다.
- **마일스톤 유지** — 태스크 1개짜리 마일스톤도 유효하다(개발 과정을 마일스톤 단위로 격리하고
  싶은 경우). 아래 절차로 계속한다.

태스크 여러 개가 자명하면 이 질문은 생략한다.

## backlog 모드

**CLI 없으면 중단.** 시작 전 `command -v backlog` 로 확인한다. 없으면 태스크 파일 손편집으로 폴백하지 말고(`SECTION:` 마커·ordinal·AC 포맷이 조용히 어긋난다) `bash "${K9HOME:-$HOME/kil9conf}/bootstrap/install-backlog-md.sh"` 설치를 안내한 뒤 멈춘다.

1. 기존 마일스톤·선행 태스크를 `backlog milestone list --plain`·`backlog task list --plain` 으로
   확인한다(이름 중복 방지, `--plain` 필수 — board·browser 등 인터랙티브 명령 금지).
2. 마일스톤의 목표·범위와 **태스크 분해안**(각 태스크의 완료 조건·의존·우선순위·단독실행)을
   정리하고, 애매한 점이 남으면 모두 해소될 때까지 `ask_question` 으로 인터뷰한다. 분해안
   (태스크 수·경계)도 확정 전에 사용자 확인을 받는다. 줄글 자유 입력은 쓰지 않는다.
3. 확정 내용으로 마일스톤을 먼저 만들고, 각 태스크를 `-m` 으로 배정해 생성한다:

   ```
   backlog milestone add "<이름>" -d "<목표·범위>"
   backlog task create "<제목>" -d "<배경·접근>" \
     --ac "<완료 조건 1>" [--ac ...] -m "<마일스톤 이름>" \
     [--dep task-N[,task-M]] [--priority high|medium|low] [-l solo]
   ```

   태스크마다 `--ac` 1개 이상 필수, `-m` 누락 금지. 마일스톤 안팎의 순서 제약은 태스크 `--dep` 로
   표현한다(마일스톤 자체엔 의존 필드가 없다).

생성한 마일스톤 이름과 태스크 ID·제목 목록을 알리고 끝낸다.

## 레거시(PLAN.md) 모드

대상 플랜 파일과 다음 `T-N`·`M-N` 번호는 이 SKILL.md 와 같은 디렉터리의 `plan-context.sh`
**1회 호출**로 확인한다(`primary` 가 대상 파일). 파일 탐색·개별 읽기를 따로 하지 않는다.
exit 1(플랜 파일 없음)이면 `/init-backlog` 으로 안내한다.

인터뷰(위 backlog 모드 2와 동일)를 마친 뒤, 새 마일스톤 섹션을 `### M-<다음 번호>. <이름>`
헤딩으로 태스크 목록 맨 아래에 추가하고, 그 아래 태스크를 `T-<다음 번호>` 부터 연속 번호 ·
`[ ]` 상태로 추가한다. 태스크 표준 필드(완료 조건/의존/범위 힌트/단독실행/접수)와 표기 규칙은
`../add-task/SKILL.md` 의 레거시 모드 절을 그대로 따르되, 마일스톤 내부 순서도 `의존:` 으로
표현한다. 기존 플랜에 마일스톤 구분이 없으면 이번 섹션부터 도입하고 기존 항목은 재편하지 않는다.
그 외 기존 내용은 수정하지 않는다.

추가한 마일스톤 번호·이름과 태스크 번호·제목 목록을 알리고 끝낸다.
