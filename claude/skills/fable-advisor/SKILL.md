# Fable Advisor

사용자가 메시지에 `/fable-advisor` 를 붙이면, **Fable 5 모델을 자문역(advisor)으로** 불러 이번 요청에
관점을 하나 더 얹는다. Fable 5 는 Agent 도구를 `model: "fable"` 로 띄운 서브에이전트다.

**자동 호출 금지.** 이 스킬은 사용자가 `/fable-advisor` 를 직접 적었을 때만 동작한다. 계획·설계·중대한
결정이라는 이유만으로 스스로 붙이지 않는다.

## 0. 무엇을 자문할지 확정

`/fable-advisor` 는 보통 요청 끝에 붙는다. **자문 대상**을 먼저 정한다:

- 메시지에 다른 내용이 같이 있으면 → 그 요청.
- 메시지가 `/fable-advisor` 뿐이면 → 직전 대화에서 다룬 가장 실질적인 것(내가 방금 낸 계획·답안,
  또는 사용자가 마지막으로 물은 것).
- `/fable-advisor` 뒤에 텍스트가 붙어 있으면 그건 자문 초점 힌트로 쓴다(예: `/fable-advisor 성능 관점으로`).

## 1. 모드 선택 (상황에 맞춰 판단)

세 모드 중 상황에 맞는 것을 고른다. 애매하면 **검토**가 기본이다.

- **검토·비평 (기본)** — 내가 먼저 구체적인 답/계획/초안을 만든 다음, 그걸 Fable 5 에 넘겨 허점·리스크·
  놓친 대안을 지적받고 반영한다. 손에 잡히는 산출물이 있을 때.
- **선조언** — 아직 착수 전이고 방향 자체가 불확실할 때. Fable 5 에게 먼저 관점을 물어 그 조언을 반영해
  진행한다. 다 만든 뒤 뒤집는 낭비를 피한다.
- **양자 병기** — 정답이 갈리는 판단·의견 문제라, 두 독립 관점을 합치지 않고 나란히 보여주는 편이
  사용자에게 더 유용할 때. 내 답과 Fable 5 의 답을 각각 라벨 붙여 제시한다.

무거운 구현 작업이면 전부 만든 뒤 검토받기보다 **계획 단계에서 선조언/검토**를 받아 되돌림 비용을 줄인다.

## 2. Fable 5 자문 스폰

서브에이전트는 내 대화를 **상속하지 않는다.** 프롬프트에 자문에 필요한 맥락을 스스로 담아라:

- 원래 요청(사용자의 목표·제약).
- 검토 모드면 내 초안/계획 전문.
- 관련 파일 경로(서브에이전트가 직접 읽도록) — 코드 전문을 붙여넣기보다 경로를 주는 편이 낫다.
- **무엇을 원하는지 명시**한다: "이 계획의 허점과 놓친 대안을 지적하라", "이 답의 사실오류·리스크를
  짚어라" 등. 막연히 "검토해줘"라고 던지지 말 것 — 그러면 맹탕 자문이 돌아온다.

스폰 규칙:

- `model: "fable"`. 이름은 글로벌 네이밍 규칙을 따르되 끝에 모델명 `-fable` 을 붙인다(예: `suisei-fable`).
- 기본은 **서브에이전트 하나.** 여러 렌즈(보안/성능/사용성 등)가 정말 필요할 때만 병렬로 늘린다 —
  자문 하나에 Fable 을 무더기로 띄우지 않는다.
- 서브에이전트의 최종 보고는 사용자에게 보이지 않는다. 받은 자문을 내가 요약·전달해야 한다.

## 3. 반영과 전달

- Fable 5 의 자문을 **무비판 수용하지 말 것.** 동의/이견을 내 판단으로 가려 반영한다.
- 무엇을 자문받아 무엇이 바뀌었는지 사용자에게 짧게 밝힌다
  (예: "Fable 자문: …. 그중 …를 반영, …는 이견이라 유지").
- 양자 병기 모드면 두 관점을 라벨을 붙여 나란히 제시한다.
- 대상이 정말 사소하면 자문으로 무겁게 만들지 말고, 한 줄로 "자문 불필요"를 밝히고 넘어가도 된다.
