# Critical/High 재검증 방법론 (공유 참조)

`/scan-bugs`(코드베이스 전수)와 diff·PR 딥리뷰 스킬이 공유한다. 실체는
`scan-bugs/severity-reverify.md` 이고 다른 소비 스킬은 상대 심링크로 참조한다 — 이 판정 기준이
두 곳에 복사되면 한쪽만 고쳐져 어긋난다.

## 왜 하는가

서브에이전트는 **자기 분야 항목을 최상 등급으로 올리는 인플레 경향**이 있다. 종합을 사용자에게
보여주기 전에, 각 Critical/High 항목의 보고된 `파일:라인`(필요하면 호출자·피호출자까지)을
**메인 에이전트가 직접 다시 읽어** 판정한다. 서브에이전트의 주장을 근거 없이 승격시키지 않는다.

## 판정 기준

- **유지**: 코드 경로와 트리거 입력이 명확하고 임팩트가 라벨에 부합한다
  (보안 침해 / 데이터 손실 / 서비스 다운 등).
- **격하(Medium/Low)**: 아래 중 하나에 해당한다.
  - 구체적 충돌 입력·경로를 코드에서 확인하지 못했다 (이론상 가능성만 있다).
  - 다른 가드가 이미 동일한 결과를 막고 있다.
  - 의도된 설계이거나 기존 동작의 포팅이다.
  - 라이브러리가 자동으로 mitigate 한다.
  - 요청당 1회·micro-allocation 수준이라 실 임팩트가 미미하다.
  - 추가 조건 탓에 실용적 공격이 어려운 베스트 프랙티스 이슈다.
- **제거**: 직접 읽어보니 서브에이전트의 코드 해석이 틀렸다.

## 진행 표시와 보존

항목마다 아래 형식으로 진행을 표시한다.

```
검증 중: {제목} → 유지(Critical)|{severity} 격하|제거 — {1줄 사유}
```

**격하·제거 항목도 삭제하지 않고 사유와 함께 결과물에 보존한다.** 무엇이 왜 내려갔는지가
남아야 다음 사람이 같은 항목을 다시 올리지 않는다. 결과물의 형태(보고서·마커·종합 출력)는
소비 스킬이 각자 정한다.
