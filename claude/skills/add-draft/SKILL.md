추가할 아이디어: `$ARGUMENTS`

아직 착수하지 않을, 나중에 정식 태스크로 편입할 후보를 보류 항목으로 기록한다. 승격(draft/아이디어 →
태스크)은 이 스킬이 아니라 `/add-task` 의 몫이다.

**모드 판별.** repo 루트에 `backlog/` 디렉터리(`backlog/config.yml`)가 있으면 **backlog 모드**,
없으면 **레거시(PLAN.md) 모드**로 동작한다.

플랜 파일도 `backlog/` 도 없는 새 저장소면 `Skill("init-backlog")` 로 먼저 초기화하도록 안내한다.

## 하지 않는 것 (양 모드 공통)

- 구현·커밋을 하지 않는다.
- `AskUserQuestion` 인터뷰를 하지 않는다(아이디어는 보류가 기본 — 정보가 부족해도 그대로 기록만 한다).
- 완료 조건(AC)을 강제하지 않는다. 착수 시 상세 인터뷰에서 채운다.

## backlog 모드

**CLI 없으면 중단.** 시작 전 `command -v backlog` 로 확인한다. 없으면 draft 파일 손편집으로 폴백하지 말고(`SECTION:` 마커·ordinal·AC 포맷이 조용히 어긋난다) `bash "${K9HOME:-$HOME/kil9conf}/bootstrap/install-backlog-md.sh"` 설치를 안내한 뒤 멈춘다.

사용자 구상을 **손실 없이** 담아 draft 로 기록한다(승격 시 인터뷰·구현의 출발점이 되도록).

```
backlog draft create "<짧은 제목>" \
  -d "<사용자 구상 무손실 기록>

접수: <오늘 날짜>. 착수 전 상세 인터뷰 필요."
```

명백한 기술적 충돌·전제(예: 순수 클라이언트 정적 사이트라 서버·API 키가 없음 등)가 보이면 `-d` 본문에
"착수 시 검토 필요: <메모>" 를 덧붙인다. 항목을 제대로 적을 만큼만 가볍게 조사한다(관련 파일 확인 정도).

생성한 draft 의 ID·제목을 알리고 끝낸다.

## 레거시(PLAN.md) 모드

대상 플랜 파일과 다음 `I-N` 번호는 `bash ~/.claude/skills/add-draft/plan-context.sh` **1회 호출**로
확인한다(`primary` 가 대상 파일, `next=I-N` 이 부여할 번호). Glob/Grep 으로 따로 찾지 않는다.
exit 1(플랜 파일 없음)이면 `Skill("init-backlog")` 로 안내한다.

1. PLAN.md 에 `## 아이디어 (보류 — 착수 전 상세 인터뷰)` 섹션이 있는지 확인한다.
   - **없으면**: `## 결정 이력` 위에 새로 만든다. 섹션 머리말로 다음을 적는다: "(아이디어)로 접수된
     항목은 `I-N`으로 별도 넘버링해 여기 모은다. 구현에 착수하지 않으며, 태스크 목록(T-N)·소비 스킬
     진행 대상이 아니다. 사용자가 착수를 지시하면 상세 인터뷰를 거쳐 정식 태스크(T-N)로 편입한다."
2. 헬퍼가 계산한 `next=I-N` 번호를 새 항목에 부여한다(중복·재사용 금지 — 헬퍼가 최대값+1 을 계산한다).
   플랜의 ID 표기가 비표준이면 헬퍼 numbers 대신 본문 표기를 이어서 따른다.
3. 아이디어를 `### I-N. <짧은 제목>` 헤딩으로 섹션 맨 아래에 추가한다. 본문에는:
   - 접수 날짜와 "착수 전 상세 인터뷰 필요".
   - 사용자가 준 구상을 **손실 없이** 기록한다(필요하면 하위 글머리로 상세를 남긴다).
   - 명백한 기술적 충돌·전제가 보이면 "착수 시 검토 필요" 메모로 덧붙인다.
4. 그 외 기존 PLAN.md 내용은 수정하지 않는다.

마지막으로 추가한 아이디어의 `I-N`·제목을 알리고 끝낸다.
