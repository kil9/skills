작업 설명: `$ARGUMENTS`

이 스킬은 backlog 를 쓰지 않는 저장소를 위한 **레거시 계획 수립 도입점**이다. 단일 `PLAN_<slug>.md` 마스터 문서를 만든다. backlog 백엔드로 계획을 짜려면 대신 `$init-backlog` 을 쓴다(그쪽이 기본). 이미 `backlog/` 가 있는 저장소에서는 이 스킬을 쓰지 말고 `$add-task`·`$init-backlog` 을 쓴다.

조사로 메워지지 않는 디테일(문제 정의, 목표·비목표, 접근 방식, 작업 범위, 제약, 테스트 전략, 우선순위 등)은 의문이 없어질 때까지 집요하게 `request_user_input` 으로 사용자를 인터뷰한다. 추측으로 채우지 않고, 줄글 자유 입력도 쓰지 않는다.

확정된 내용으로 현재 디렉토리에 `PLAN_<slug>.md`(소문자 케밥케이스 2\~4단어) 를 작성한다. 세션 리셋 후에도 이 파일만으로 재개 가능할 만큼 상세히 적고, 배경/문제/목표/비목표/제약, 접근 방식과 근거, 태스크 목록, 검증 방법, 마지막 갱신 날짜를 포함한다. 같은 파일이 있으면 덮어쓸지 확인한다.

**넘버링 규칙 (M / T / I).**
- `M-N` 마일스톤: 태스크를 묶는 상위 단위 (`M-1`, `M-2` …). 규모가 작아 묶을 필요가 없으면 마일스톤 없이 태스크만 나열해도 된다.
- `T-N` 태스크: 마일스톤 아래의 실행 단위 (`T-1`, `T-2` …), 상태는 아래 상태 표준을 따른다. **플랜 전체에서 유일하게 증가하는 전역 카운터**(중복·재사용 금지, 마일스톤이 바뀌어도 리셋하지 않는다).
- `I-N` 아이디어: 보류 항목 (`I-1`, `I-2` …), 별도 카운터.

**태스크 상태 표준.** 상태는 아래 4개만 쓴다. 이 정의가 레거시 플랜 스킬군의 원본이며(소비자 스킬은 상태별 행동만 각자 정의한다), 새 상태를 임의로 추가하지 않는다. 기존 플랜의 비표준 상태 표기는 강제 변환하지 않는다(커스텀 ID 표기 존중과 동일 원칙).

- `[ ]` TODO: 착수 전.
- `[→]` IN_PROGRESS: 진행 중.
- `[x]` DONE: 검증·커밋까지 완료.
- `[!]` BLOCKED: 사람의 개입·외부 요인 해소(사용자 결정 대기, 외부 시스템·권한 등) 없이는 진행 불가. 상태 라인에 사유 한 줄을 반드시 병기한다. 재실행으로 풀릴 수 있는 단순 자동 실행 실패는 BLOCKED 가 아니라 TODO 유지 + 실패 로그 append 다.

**태스크 표준 필드.** 각 `T-N` 태스크 본문에는 다음 필드를 반드시 적는다. 소비자 스킬(`$start-task`·`$parallel-tasks` 의 레거시 모드)이 이 필드로 검증 기준과 병렬 안전성을 판단한다.

```markdown
### T-N. <제목>
- 완료 조건: <검증 가능한 기준>
- 의존: T-x (선행 태스크가 없으면 "없음" 을 반드시 명시)
- 범위 힌트: <변경 대상 파일·모듈> (파악된 경우만)
- 단독실행: 필요 — <사유> (다른 태스크와 병렬 실행이 안전하지 않은 경우만 적는다. 생략 = 병렬 허용)
- 접수: YYYY-MM-DD
```

아직 착수하지 않을 후보 아이디어가 있으면 태스크(`T-N`)로 넣지 말고 별도 `## 아이디어 (보류)` 섹션에 `I-N` 으로 적는다(start-task 진행 대상 아님, 착수 지시 시 상세 인터뷰 후 태스크로 편입). 아이디어 추가·관리는 `$add-draft` 스킬을 쓴다.

PLAN 경로와 태스크 목록을 보고하고 **멈춘다. 구현은 시작하지 않는다**(생산자 원칙: 이 스킬은 계획 수립까지만 담당한다). 소비는 별도 세션의 `$start-task`(순차)·`$parallel-tasks`(worktree 병렬) 레거시 모드 몫임을 안내한다. 사용자가 이 세션에서 바로 진행하라고 명시적으로 지시할 때만 `$start-task` 절차(구현 → 검증 → 커밋을 태스크 단위로 완결)로 이어간다.
