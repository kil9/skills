name: next-backlog
description: backlog(또는 PLAN.md)만 훑어 마일스톤별로 남은 태스크를 나열하고 다음에 착수할 것 하나를 추천한다. 추가 조사·구현은 하지 않는다. "다음 뭐 하지 / 남은 일 뭐 있지 / 어디까지 왔지 / 전체 진척 / 다음 마일스톤" 을 물을 때 사용한다.
---

백로그(또는 플랜 파일)만 읽고 **마일스톤을 단위로 남은 태스크를 나열**한 뒤 다음에 착수할 것 하나를 추천한다. 진척 개괄(어디까지 왔는지)과 태스크 후보(무엇을 할지)를 한 화면에서 같이 본다.

**읽기 전용**이다: 태스크·플랜·코드를 건드리지 않고 보고만 한다. 근거는 backlog 데이터(레거시 모드는 플랜 본문)뿐 — 코드·이슈·git·웹은 조사하지 않는다. 후보를 고른 뒤 실제로 하는 것은 `$start-backlog`·`$loop-backlog` 몫이다.

## 0. 호출 인자 분기

- **인자 없음** → 기본. §1-4 대로 전체 대시보드 + 마일스톤별 남은 태스크.
- **마일스톤 id**(`m-4` 등) → 그 마일스톤 하나만. 대시보드를 생략하고 §5-1 대로 그 안의 남은 태스크만 상세히 낸다.
- **`--tasks`** → 마일스톤 묶음 없이 §5-2 대로 태스크 후보만 평평하게 나열한다.

인자가 마일스톤 id 로도 `--tasks` 로도 해석되지 않으면(오타 등) 임의로 짐작해 진행하지 말고, 인자를 무시한다고 밝힌 뒤 기본 동작을 한다.

## 1. 모드 감지

repo 루트에 `backlog/` 디렉터리(또는 `backlog/config.yml`)가 있으면 **backlog 모드**(§2-5), 없으면 **레거시 모드**(§6)로 동작한다. backlog 모드가 기본이다.

**CLI 없으면 중단.** backlog 모드면 `command -v backlog` 로 확인한다. 없으면 태스크 파일을 직접 읽어 폴백하지 말고(상태·의존·마일스톤을 잘못 읽는다) `bash "${K9HOME:-$HOME/kil9conf}/bootstrap/install-backlog-md.sh"` 설치를 안내한 뒤 멈춘다.

## 2. 수집 (backlog 모드)

### 2-1. 마일스톤

```
backlog milestone list --plain --show-completed
```

`--plain` 은 필수다(보드/브라우저 실행 금지). `--show-completed` 가 없으면 완료 마일스톤이 `(collapsed, use --show-completed to list)` 로 접혀 대시보드에 안 나온다 — 전체 진척이 목적이므로 항상 붙인다.

출력은 두 그룹이다.

```
Active milestones (5):
  m-1: skills repo 분리·공개 (0/1 done)
  m-4: F13 STT 재부팅 내성 (5/6 done)

Completed milestones (2):
  m-6: vial 치트시트 인쇄 조판 개선 (7/7 done)
```

각 줄에서 **id·제목·완료/전체 카운트**를 얻는다. 이 한 번의 호출로 진척 바에 필요한 건 다 나온다 — 카운트를 얻으려고 태스크를 세지 말 것.

마일스톤이 하나도 없으면 대시보드를 포기하고 `--tasks` 와 같은 출력(§5-2)을 내면서 `$add-milestone` 을 안내한다. 마일스톤이 없다는 이유로 멈추지는 않는다 — 남은 태스크를 아는 것이 원래 목적이다.

### 2-2. 태스크

```
backlog task list --plain
```

`--plain` 은 필수다. 출력은 상태별 그룹(`To Do` / `In Progress` / `Done` / `Blocked`)으로 나오고, 각 줄은 `[PRIORITY] TASK-N - 제목` 형식이다(priority 미설정 시 앞의 대괄호 없음). draft·milestone 은 여기 안 나오므로 **자동 제외**된다.

여기서 **Done 태스크 ID 집합**(의존 해소 판정용)과 미완료(To Do / In Progress / Blocked) 태스크의 ID·제목·priority 를 얻는다.

태스크가 하나도 없으면 그 상태를 보고하고 `$init-backlog` 을 안내한 뒤 멈춘다.

### 2-3. 미완료 태스크 상세

목록만으로는 소속 마일스톤·의존·label·AC 가 안 보이므로 **미완료 태스크만** 개별 조회한다. Done·draft 는 조회하지 않는다 — 카운트는 §2-1 이 이미 줬다.

```
backlog task <id> --plain
```

`Milestone:` / `Dependencies:` / `Status:` / `Labels:` / `Acceptance Criteria:` / `Implementation Notes:` 를 읽는다. 파싱에서 걸리는 것 둘:

- **값이 없는 필드는 줄 자체가 안 나온다.** 마일스톤·의존·label 이 없는 태스크에는 `Milestone:` 줄이 아예 없지 `Milestone: (none)` 같은 게 아니다. 줄이 없으면 "미설정"으로 읽을 것 — 조회 실패로 오해해 재시도하지 말 것.
- **`Status:` 값에는 글리프가 붙는다**(`○ To Do` / `◒ In Progress` / `● Blocked` / `✔ Done`). 상태를 문자열 동등 비교하면 전부 어긋나므로 글리프를 떼고 본다. 상태의 진실원본은 이쪽이며, §2-2 목록의 그룹 헤더와 같은 값이다.

## 3. 착수 가능 / 막힘 판정

- **In Progress** → 착수 가능(이어서 할 일).
- **To Do** 이고 의존이 비었거나 나열된 ID 가 모두 Done 집합에 속함 → 착수 가능.
- **To Do** 인데 의존이 안 풀림 → `막힘(task-x 대기)`.
- **Blocked** → 막힘. 사유는 `Implementation Notes:` 첫 줄을 따 `막힘(사유)`.
- `Labels: solo` 가 있으면 `[solo]` 를 병기한다(단독실행 — `$parallel-tasks` 선택에 영향).

진척 바만으로는 **막힌 마일스톤과 그냥 안 한 마일스톤이 구별되지 않는다**. 5/6 인데 남은 하나가 사용자 입회 대기라면 추천해봐야 소용없으므로, 이 판정은 생략하지 않는다.

## 4. 기본 출력

완료된 마일스톤까지 포함해 **전부** 나열하고, 미완료 마일스톤 아래에 그 마일스톤의 **남은 태스크를 전부** 들여쓴다.

```
## 전체 진척

[########--] m-4  F13 STT 재부팅 내성  (5/6)
    task-117  토스트 지연 완화        막힘(사용자 입회)

[----------] m-7  herdr 에서 키맵      (0/1)
  → task-121  키맵 뷰어 붙이기        착수 가능 [solo]

[##########] m-6  vial 치트시트 조판    (7/7)  완료

마일스톤 없음: task-130, task-131 (둘 다 착수 가능)

← 추천: task-121 (m-7) — 의존 없고 단독 실행 가능.
```

- 진척 바는 10칸 고정. 완료율을 반올림해 채운다. 전체가 0 이면(`0/0`) 빈 바로 두고 `태스크 없음` 을 붙인다 — **0 으로 나누지 말 것**. 마일스톤은 태스크 없이 먼저 만들어지는 일이 흔해서 이 경우가 실제로 자주 나온다.
- 태스크 줄은 **착수 가능이면 `  → `, 막힘이면 공백 4칸**으로 시작한다. 화살표 유무가 "지금 손댈 수 있는가"의 유일한 시각 신호다.
- 완료 마일스톤은 태스크를 펼치지 않는다(남은 것이 없다). `완료` 꼬리표만 붙인다.
- 마일스톤에 속하지 않은 미완료 태스크는 대시보드 아래 별도 줄로 모은다. 개수가 많으면 착수 가능한 것만 나열하고 나머지는 `외 N개 막힘` 으로 줄인다.
- 정렬은 진척 높은 순이되 완료 마일스톤을 위로 몰지 말 것(전체를 한눈에 보는 게 목적이다). 완료분은 `완료` 꼬리표로 구분되므로 순서로 또 가를 필요가 없다.
- **제목 열을 문자 수로 정렬하지 말 것.** 한글 제목은 한 글자가 터미널 2셀이라 `%-32s` 류로 맞추면 제목 길이에 따라 뒤 열이 들쭉날쭉해진다. 셀 폭으로 세거나(East Asian Wide/Fullwidth = 2), 아예 고정폭 정렬을 포기하고 공백 하나로 이어 붙인다. 후자가 안전하다 — 대시보드의 값은 정렬이 아니라 한눈에 들어오는 것이다.

**추천 하나**를 맨 아래 붙인다. 고르는 순서:

1. **In Progress 태스크** — 이미 손댄 것을 먼저 끝낸다.
2. 착수 가능 태스크 중, **진척이 가장 높은 미완료 마일스톤**에 속한 것(끝내기 가장 가까운 덩어리). 같은 마일스톤 안에서는 priority(High > Medium > Low, 미설정은 최하) 순.
3. 마일스톤 소속 태스크가 하나도 착수 가능하지 않으면 마일스톤 없는 태스크에서 고른다.
4. 그것도 없으면 추천하지 않고 "전부 막혔거나 완료" 라고 알린다. 이때 막힘 사유를 태스크별로 한 줄씩 보여 준다(무엇을 풀어야 진행되는지가 그 상황에서 유일하게 쓸모 있는 정보다).

추천에는 **한 줄 근거**를 붙인다. 진척 숫자만 다시 말하지 말고 왜 지금 그것인지를 적는다.

## 5. 인자 모드 출력

### 5-1. 마일스톤 지정 (`m-N`)

대시보드를 생략하고 그 마일스톤만 낸다. 제목·진척 카운트 한 줄 뒤에 남은 태스크를 **AC 요약까지 포함해** 상세히 나열한다(기본 출력보다 한 단계 깊게 — 범위를 좁힌 이유가 그것이다). 맨 아래 추천은 그 마일스톤 안에서만 고른다.

지정한 id 가 없으면 있는 마일스톤 id 목록을 보여 주고 멈춘다. 이미 완료된 마일스톤이면 완료 사실을 알리고 태스크는 펼치지 않는다.

### 5-2. `--tasks`

마일스톤 묶음 없이 착수 가능 태스크를 평평하게 나열한다. In Progress 를 맨 앞에, 나머지는 priority 순. 항목마다 **task ID·제목·AC 요약·지금 착수 가능한 이유 한 줄**. 막힘 태스크는 후보에서 빼고 아래에 `막힘(사유)` 로 따로 모은다. 맨 앞 하나를 추천한다.

## 6. 레거시 모드 (PLAN.md)

`backlog/` 가 없으면 이 스킬 디렉터리의 `plan-context.sh --dump` 를 셸 명령 1회로 실행해 플랜 파일 전문을 받는다.

```
bash ~/.codex/skills/next-backlog/plan-context.sh --dump
```

exit 1(플랜 파일 없음)이면 알리고 `$init-backlog` 을 안내한 뒤 멈춘다.

표기가 표준 `T-N`/`I-N`/`M-N` 이 아닐 수 있다 — 근거는 본문뿐이다.

- `M-N` 아래 딸린 `T-N` 들의 `[x]`/`[ ]`/`[→]`/`[!]` 로 완료·착수가능·막힘을 세어 §4 와 같은 대시보드를 만든다.
- `[→] IN_PROGRESS` 는 이어서 할 일로 맨 앞. `[ ] TODO` 중 선행 의존 미완은 `막힘(T-x 대기)`. `[!] BLOCKED` 는 `막힘(사유)`.
- `단독실행: 필요` 필드가 있으면 병기한다.
- `I-N` 자체는 제외한다. 플랜 파일이 여럿이면 파일별로 묶어 밝힌다.
- **`M-N` 구분이 아예 없거나 표기가 제각각이면 억지로 만들지 말 것.** 파일이 마일스톤 개념 없이 태스크만 나열하는 경우가 흔하다. 그때는 마일스톤 구분이 없다고 밝히고 `--tasks` 와 같은 평평한 출력(§5-2)으로 낸다. 태스크를 임의로 묶어 가짜 마일스톤을 만들면 없는 구조를 있는 것처럼 보고하게 된다.

## 7. 경계

- 읽기 전용이다. `backlog task edit`·`milestone edit` 류를 부르지 않는다. 상태가 틀려 보여도 고치지 말고 보고만 한다.
- 여기서 멈춘다 — 추천했다고 이어서 구현하지 않는다. 그건 `$start-backlog`·`$loop-backlog` 몫이다.
