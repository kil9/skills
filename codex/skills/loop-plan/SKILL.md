레거시 `PLAN*.md` 계획을 **스스로 진행할 수 있는 태스크가 남지 않을 때까지** 라운드 단위로 자율 드레인한다(backlog 백엔드는 `$loop-backlog`). 각 라운드: **fresh 재조회 → ready set → 실행 → 발견분 자동 추가**. 각 태스크는 구현 → 검증 → 커밋(`$start-task` 레거시 규칙)으로 완결한다. **한 번 읽고 비었다고 종료하지 않고**(§5), **인터랙티브 질문으로 멈추지 않는다**(§3).

## 0. 전제 · 대상 파일

대상은 현재 디렉터리의 `plan*.md`(대소문자 무관 — 관례상 `PLAN.md`·`PLAN_*.md`). 여러 개면 **모두** 대상으로 삼고 어느 파일들인지 밝힌다. 하나도 없으면 "backlog 있으면 `$loop-backlog`, 아니면 `$make-a-plan`" 안내 후 중단.

상태 `[ ]` TODO / `[→]` IN_PROGRESS / `[x]` DONE / `[!]` BLOCKED. 실행 단위는 `T-N`, `M-N`(묶음)·`I-N`(아이디어)은 진행 대상이 아니다. 비표준 ID 표기(예: `T1`, `NU-3`)는 존중한다 — 근거는 본문뿐이다.

## 1. 라운드 · ready set

라운드 시작마다 대상 `plan*.md` 를 **다시 읽는다**(이전 라운드에서 읽은 내용 재사용 금지 — 외부 세션·사용자가 라운드 중 편집한 내용을 흡수한다).

ready set = 지금 스스로 착수 가능한 태스크:

- `[→]` IN_PROGRESS(이어서 진행 — 최우선).
- `[ ]` TODO 중 의존(`의존:` 필드, 없는 구식 플랜은 본문의 선행 관계)이 모두 `[x]` 이고 **self-actionable**(진행에 사용자의 중대한 결정이 필수가 아님 — §3).
- `[!]` 는 제외하되, 막힘 사유가 해소된 게 확인되면 `[ ]` 로 되돌려 포함. `I-N`·`M-N` 은 제외.

## 2. 실행 (솔로 기본 · 병렬은 게이트 통과 시만)

**기본은 솔로 순차.** ready set 을 우선순위·의존 순으로 한 태스크씩 `$start-task` 레거시 루프(구현 → 검증 → 커밋, 커밋 전 상태를 `[x]` 로 갱신해 같은 커밋에)로 완결하고 §3·§4 를 얹는다.

병렬은 `$parallel-tasks` 의 병렬 적합성 게이트(워커 cold-start·낮은 실효 동시성·토큰 배수의 함정)를 **인지하고도** 이득이 분명할 때만 쓴다: substantial 독립 태스크(다파일·신규·리팩터급이고 서로 파일이 안 겹침, `단독실행: 필요` 없음) ≥ 2개일 때. 애매하면 솔로. 선택 사유를 한 줄 보고한다.

- 병렬 라운드는 게이트를 통과하는 태스크만 `$parallel-tasks` 레거시 모드(워커는 코드만, PLAN.md 는 리드가 머지 단계에서 일괄 갱신)로, 나머지 사소·의존 태스크는 리드가 인라인 처리한다. 워커 프롬프트에 "발견한 선행·후속 작업을 RESULT 로 보고"를 넣고 리드가 머지 후 §4 로 태스크화한다.

## 3. 하이브리드 막힘 정책 (질문으로 멈추지 않는다)

- **사소한 결정**(네이밍·에러 처리·라이브러리 세부 등 되돌리기 쉬운 것): 스스로 합리적으로 정해 진행. 영향이 크면 PLAN 에 `### 디자인 결정: <제목>`(날짜·결정·이유·대안) 블록으로 기록해 커밋에 포함.
- **사용자 몫인 중대한 갈림길**(결과를 크게 바꾸거나 되돌리기 어려움): 추측으로 밀지 않는다. 상태를 `[!]` 로 바꾸고 상태 라인에 `결정 필요: <질문 한 줄>` 을 병기한 뒤 다음 태스크로. self-actionable 이 아니므로 종료를 막지 않는다.
- **재실행으로 풀릴 일시적 실패**(flaky·환경): `[!]` 아님 — `[ ]` 유지 + 진행 로그에 실패 요약 append, 다음 라운드 재시도. 같은 태스크가 반복 실패하면 그때 `[!]` 로 전이한다.

## 4. 발견분 자동 추가

기존 태스크 완료에 **실제로 필요한** 선행·후속 작업(누락된 마이그레이션, 선행 리팩터, 완료 조건을 막는 버그 등)을 발견하면 즉시 `T-N` 태스크로 추가해 다음 라운드가 집게 한다:

- 번호 = 대상 플랜들 전체에서 기존 `T-N` 최대값 + 1(전역 카운터, 재사용 금지). 적절한 마일스톤 아래 맨 밑에 `[ ]` 로.
- 표준 필드: `완료 조건:`·`의존:`(없으면 `없음`)·`범위 힌트:`(파악된 파일만)·`단독실행:`(병렬 비안전 시만 `필요 — <사유>`)·`접수:`(오늘). 본문 첫 줄에 `loop-plan 자동 추가: <발견 맥락>`.
- **scope creep 경계**: "있으면 좋은" 개선은 `T-N` 이 아니라 `## 아이디어 (보류)` 의 `I-N` 으로(ready set 밖).
- 자동 추가분의 완료 조건 자체가 사용자 결정을 요하면 만들되 §3 대로 `[!]` 로 둔다.

## 5. 종료 (연속 2회 빈 조회 필수)

ready set 이 비어도 **즉시 종료하지 않는다** — 외부 세션·사용자가 방금 태스크를 추가·편집했을 수 있다. 빈 라운드 뒤 §1 로 **한 번 더 다시 읽어** 그것도 비었을 때만 종료한다(연속 2회 빈 조회). 새 태스크가 생겼으면 라운드를 재개한다.

**수렴 안전장치**: 자동 추가(§4)가 새 태스크를 계속 만들어 미완료가 줄지 않으면(예: 3라운드 연속 완료 0 · 신규만 증가) 무한 루프로 보고 멈추고 남은 태스크를 사용자에게 알린다.

## 6. 마무리

PLAN 은 세션 리셋 후에도 재개 가능한 진실원본이다 — 라운드/커밋 경계에서 세션 리셋 후 이 스킬을 다시 호출해도 손실이 없다. 종료 시 보고: 완료(파일·ID·제목)·자동 추가·`[!]`(사유·질문 포함) 태스크, 기록한 설계 결정, 종료 근거. 완료(`[x]`) 태스크가 5개 이상이면 작업 ID·한 줄 요약만 남기도록 압축을 한 줄로 권고한다.
